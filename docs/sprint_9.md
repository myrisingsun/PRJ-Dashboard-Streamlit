# Sprint 9 — Результаты

## 1. Исправление парсера финансового листа (loader.py)

### Проблема: лишние строки попадали в чужие проекты

В `load_prj_money()` строки с пустым col 0 (статья без кода проекта), идущие
после блока проекта, forward-fill'ились под последний активный `current_code`.
Например, строки "PLAN R" попадали в CSDC.0001, "БАЗИС" — в SALE.0005.

### Исправления в `data/loader.py`

**Фикс 1 — пустая строка сбрасывает проектный блок:**
```
Пустая строка (разделитель) → current_code = "" → continue
```
Раньше пустые строки просто пропускались (`continue`) без сброса `current_code`.

**Фикс 2 — непустой col 0, не совпадающий с regex кода проекта, сбрасывает блок:**
```
c0 непустой и не ^[A-Z]{2,6}\.\d{4}$ → current_code = ""
```
Строки, промаркированные пользователем "отличным индексом" (не-проектным кодом),
больше не попадают в предыдущий проект.

## 2. Финансовые KPI-карточки — единый источник данных (3_Finance.py)

### Проблема

KPI-карточки вверху страницы считались из `get_finance_per_project()`
(отдельная функция с отдельной логикой), а таблицы ниже — из `load_prj_money()`.
Два источника могли давать расхождение, особенно при фильтрации проектов.

### Исправление

- `filtered_money` и `ordered_codes` вычисляются **до** KPI-блока
- KPI считаются из `filtered_money[is_итого]` — тех же данных, что и таблицы
- Импорт `get_finance_per_project` и загрузка `fin_totals` удалены
- `make_portfolio_chart()` переработан: данные берутся из `filtered_money`
  с группировкой по `Код проекта`, а не из отдельного источника

Результат: сумма наверху страницы всегда равна сумме строк «Итого» в таблицах
ниже и реагирует на фильтр проектов идентично.
